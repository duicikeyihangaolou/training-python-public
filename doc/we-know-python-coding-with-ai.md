**1.2.3 AI 编程工具选型与接入**

在 AI 时代，编辑器不只是“写代码的地方”，更是**指挥、调用 AI 来一起编程的控制台**。建议：

- 至少熟悉一种“内置在 IDE 里的”编码助手：
  - Cursor / GitHub Copilot / Claude Code / JetBrains AI 等；
- 知道它们擅长和不擅长的事（可参考 `we-know-python.md` 中的“AI 时代的 Python 学习”小节）：
  - 擅长：补全样板代码、改写 / 重构、生成测试、写文档；
  - 不擅长：理解你复杂的业务隐性约束、替你做架构决策、独立承担上线风险。

AI 浪潮之下，技术更新迭代令人目不暇接。但另一方面，要理清这其中变与不变的东西，以此为判断依据，了解 AI 适用于哪些场景，不适合哪些场景。这样才不会被焦虑侵袭，也能恰如其分地发挥 AI
工具的价值。

有如下几条供参考阅读：

1. AI 的本质没有变，依然是 80 年代末已经发展到完备的神经网络理论（连接主义、大数定理），其上限被样本数据锁定，下限被算力锁定。因此 AI
   可以通过强化学习下围棋下赢人类，但对于强化学习不适用的场景就无能为力。基于神经网络的 AI
   不可能写出比《将进酒》更好的诗；也突破不了人类现有的知识边界（比如从牛顿经典力学突破到量子力学、相对论）。
2. 所有可实现的
   AI，本质都是图灵完备的形式系统，符合哥德尔不完备性定理推论（封闭系统无法从内部证明自身一致性，也不能从内部彻底替换自身公理体系）。只有人类大脑可以拥有第一人称主观视角，可以站在当前认知系统之外，对系统进行观察、批判、重构和替换。这是人类和纯形式主义在结构上的根本差异。
3. 编程本质上是符号主义学习类型，而现在的 AI 是连接主义。人类的大脑学习则远不止连接主义。因此即便在软件行业上，AI 也无法完全代替人类。发明打字机后，好书并没有多多少；用 AI
   写公众号文章，十万加的文章并没有多多少；用 AI 编程，现在看起来，好的软件也没有多多少。马斯克和奥特曼说的 AGI 可能很快会来，但 Google 认为的 AGI 则遥遥无期（除非当下 AI
   的本质被突破，不再是基于神经网络的形式系统）。

我们该怎么办？

1. 对基础理论，应知其所以然。在具体做事的时候，外行领导不了内行，因此要放权。管理人类员工是这样，管理 AI 数字员工亦然。人月神话，在 AI 时代依然适用。
2. 善用 AI 工具，知道怎么造轮子就可以了，不要自己去造，让 AI 去做这些繁而不难的细节事，把注意力用来宏观思考软件工程的整体面。

### 1.3 动态类型与对象模型

[返回目录](#课程目录)

本节主要承接 `we-know-python.md` 中的：

- [1.3 变量和对象](we-know-python.md#13-变量和对象)
- [1.3.3 对象的可变性](we-know-python.md#133-对象的可变性)
- [1.3.4 在 PVM 中的存储模型](we-know-python.md#134-在-pvm-中的存储模型)
- [1.4 基本对象类型](we-know-python.md#14-基本对象类型)（列表 / 元组 / 集合 / 字典）

并进一步强调**实战中最容易踩坑的几类问题**。

**1.3.1 变量、引用与对象（三件事）**

- 变量是什么？——只是**名字 / 引用**，本身没有类型。
- 对象是什么？——真正占内存、有类型和值的实体。
- 赋值 `a = b` 的含义：**让变量 `a` 指向和 `b` 相同的对象**，并不会复制对象本身。

典型结论（来自 `we-know-python.md`）：

- “变量只有直接位于等号左边的时候，表示为变量建立映射关系；其他时候都表示其映射到的对象本身。”
- 没有被引用的对象，会被 GC 回收。

**1.3.2 可变 / 不可变对象与函数参数坑**

结合课程大纲中的“可变 / 不可变类型在函数参数、容器中的差异”，这里重点区分：

- **不可变对象**：`int / float / str / tuple / frozenset` 等
  - 值相等的对象往往会在 PVM 中共享一份内存；
  - 修改“值”实际上是创建了一个新对象，并重新绑定变量。
- **可变对象**：`list / dict / set` 等
  - 就地修改（如 `list.append`、`dict.update`）会影响所有持有该引用的变量。

高频坑示例（伪代码形式，后续可在练习中展开）：

- 把可变对象作为**默认参数**；
- 在函数内部“修改参数”，结果影响了外部逻辑；
- 把可变对象放进多个容器（如 `a = [[0]] * 3` 等例子，可参考 `we-know-python.md` 中对应思考题）。

**1.3.3 对象模型与调试技巧**

为了更好地理解和调试对象状态，本节鼓励你熟练使用：

- `id(obj)`：查看对象“身份”（是否真的是同一个对象）。
- `type(obj)` / `isinstance`：区分类型和继承关系。
- `__dict__` / `dir(obj)`：观察一个对象当前拥有的属性。

第一天的练习会引导你：

- 画出简单的“变量 → 对象 → 容器”内存示意图；
- 对比几段看似一样、实则对象共享或拷贝行为不同的代码；
- 用 `id()` / `is` / `==` 来验证自己的理解。

### 1.4 函数与作用域进阶

[返回目录](#课程目录)

本节承接 `we-know-python.md` 的：

- [2.1 函数和高阶函数](we-know-python.md#21-函数和高阶函数)

重点**不是语法罗列**，而是围绕实战高频 bug，理解“避免踩坑的函数使用原则”。

**1.4.1 函数定义与调用的本质**

- 定义函数的过程：
  1. 解释器创建一个函数对象；
  2. 把函数对象绑定到函数名（变量）上；
  3. 多次定义同名函数，后者会覆盖前者。
- 调用函数的本质：
  - 把实参对象映射到形参变量；
  - 在新的局部命名空间中执行函数体；
  - 可选地返回一个对象，再绑定到调用方的变量。

**1.4.2 LEGB 作用域规则与常见陷阱**

LEGB：`Local → Enclosed → Global → Built-in`

- 在函数内部出现“赋值”的变量，默认都会被认为是 **Local**，哪怕你只是想“读”一下全局变量。
- 高频错误示例（简化自 `we-know-python.md`）：
  - 在函数内部写 `a += 1`，但 `a` 只在全局定义，未声明 `global a`，会触发 `UnboundLocalError`。

课堂会重点演示：

- 闭包中捕获外部变量、`for` 循环里定义函数时的常见问题；
- 何时该用 `global` / `nonlocal`，何时应该**干脆改成参数传递 + 返回值**，让依赖更清晰。

**1.4.3 参数类型与默认值策略**

基于 `we-know-python.md` 的内容，这里会收束出几条**实战约定**：

- 参数顺序：位置参数 → 关键字参数 → `*args` → `**kwargs`，保持接口稳定；
- 默认参数的两条铁律：
  1. 默认值在函数**定义时**计算一次；
  2. 默认值**尽量不要使用可变对象**（`list / dict / set` 等）。

推荐写法：

- 不用 `def foo(a_list=[])`，改为：
  ```python
  def foo(a_list=None):
      if a_list is None:
          a_list = []
      ...
  ```
- 当你需要“延迟计算”默认值（如当前时间、动态配置）时，优先在函数体内计算，而不是写死在默认参数里。

**1.4.3.1 类型标注（PEP 484）**

在函数形参和返回值上写类型标注，**运行时不会做类型检查**，但能提升可读性、IDE 提示和静态检查（如 mypy），便于把“能跑的小脚本”改成“敢上线的小工具”。

- 写法：`def 函数名(形参: 类型, ...) -> 返回类型:`；无返回值时写 `-> None`。
- 常用类型：`int` / `str` / `float` / `bool`；`list[str]`（Python 3.9+）或
  `List[str]`（`typing`）；`dict[str, int]` 或 `Dict[str, int]`；`Optional[T]` 或 `T | None` 表示可为 `None`。
- 示例：带类型标注的工具函数（与下面实战练习一致）
  ```python
  def parse_int(value: str, default: int | None = None) -> int | None:
      """将字符串转为整数；失败时返回 default，若 default 为 None 则抛出 ValueError。"""
      try:
          return int(value.strip())
      except ValueError:
          if default is not None:
              return default
          raise ValueError(f"无法将 {value!r} 转为整数") from None
  ```
- 可选：用 `mypy` 做静态检查（`pip install mypy`，`mypy 脚本名.py`），不改变运行时行为。详见
  [we-know-python.md 2.1.1 类型标注](we-know-python.md#211-函数的定义和调用)。

**1.4.4 实战小练习（为后续 AI 编程做铺垫）**

为了和第三天的 LLM / Agent 连接，本节的练习会引导你实现 1–2 个**带参数校验的工具函数**，例如：

- 一个安全的 `parse_int(value: str, default: int | None = None) -> int | None`：
  - 支持类型标注；
  - 对异常输入有清晰的行为（返回默认值 / 抛出明确异常）；

### 1.5 高阶函数与容器选型

[返回目录](#课程目录)

本节承接 `we-know-python.md` 中的：

- [2.1.2 高阶函数](we-know-python.md#212-高阶函数)

结合大纲中的“写少、跑快套路”，重点放在两个方面：

1. **用合适的容器表达真实意图**（列表、集合、字典各自适用场景）；
2. **用推导式 + map/filter/reduce 提升表达力，但不过度炫技**。

**1.5.1 容器选型心法**

- 列表 `list`：有序，可重复，适合“序列 / 队列 / 排序后再处理”的场景；
- 元组 `tuple`：不可变序列，适合作为“多值返回 / 不希望被篡改的配置片段”；
- 集合 `set`：去重、集合运算（交 / 并 / 差），常用于“去重统计 / 快速判断是否存在”；
- 字典 `dict`：键值映射，是“用键找值”的首选结构。

本节会通过小练习对比：

- 用列表 + 手写去重 vs 直接用集合；
- 用平铺的 `if/elif` 查找配置 vs 用字典映射（类似“命令分发表”）。

**1.5.2 推导式与生成器表达式**

在 `we-know-python.md` 介绍的基础上，本课程强调三条实践规则：

- **简单过滤 / 映射** → 用推导式（列表 / 集合 / 字典）更直观；
- **大数据量、只遍历一次** → 优先使用生成器表达式，避免不必要的内存占用；
- 一行里出现超过 2 层嵌套或复杂逻辑 → 考虑改回普通 `for` 循环，换取可读性。

示例模式（教学中会给出完整代码）：

- 清洗一批用户输入：`strip` + 过滤空字符串 + 统一大小写；
- 用列表推导式重写多行 `for` 循环，使意图更集中。

**1.5.3 高阶函数在业务代码中的合理使用**

从 `we-know-python.md` 中的 `map` / `filter` / `reduce` 出发，这里更强调“在项目里的姿势”：

- 什么时候用 `map` / `filter` 比 for 循环更好：
  - 当“转换逻辑很短”且“你已经有一个清晰的函数实现”，例如 `map(normalize_user, users)`；
- 什么时候避免 `reduce`：
  - 当新人阅读成本远大于 for 循环，但在处理“聚合所有值”的场景可以适度使用，如 `sumAny`。

本节的小练习会围绕一组“模拟日志行 / 用户数据”，分别用：

- 传统 for + if；
- 推导式；
- map / filter / reduce；

实现**相同的过滤和统计需求**，帮助你体会在可读性、性能和行数之间的权衡。

### 1.7 异常与健壮性设计

[返回目录](#课程目录)

本节直接对标 `we-know-python.md` 中的：

- [2.4 异常处理](we-know-python.md#24-异常处理)

结合课程大纲中的目标：为后两天的日志工具和 AI 应用打好“**不轻易崩溃 + 能解释发生了什么**”的基础。

**1.7.1 异常体系与捕获策略**

常见经验总结：

- 尽量捕获**具体异常类型**（如 `ValueError` / `IOError` / `ZeroDivisionError`），避免一上来就 `except Exception`；
- 不要“吞异常”：至少要记录日志或重新抛出，避免用空的 `except: pass`；
- 把“期望内的错误”（如用户输入不合法）和“程序缺陷 / 外部系统异常”区分对待：
  - 前者可以转成业务友好的提示；
  - 后者需要尽可能保留堆栈信息，方便排查。

**1.7.2 资源管理与 `with` 语句**

在文件 / 网络连接 / 数据库 / 锁 等场景，推荐统一使用上下文管理器：

- 把“打开 + 使用 + 关闭”抽象成一个 `with` 块；
- 业务逻辑里专注在“用资源做什么”，而不是重复写 try/finally。

课堂会对比两版代码：

- 传统 try/finally 写法；
- 等价的 `with` 写法；

并展示一个简化的自定义 context manager，用来包裹“带重试和日志”的操作（为后续封装网络请求、LLM 调用做铺垫）。

**1.7.3 日志与异常的联动设计**

虽然 `loguru` 等库会在后续细讲，这里先确立几个约定：

- 对“不会再往外抛”的异常，务必打出包含关键信息的日志（输入摘要 / 上下文 ID / 错误类型等）；
- 对“向上抛”的异常，可以在底层使用简洁消息，交由上层统一格式化；
- 避免在 catch 里随意改写异常类型，除非你有清晰的“错误边界层”设计。

在第一天的小结练习中，你会被要求为一个“个人信息管理小工具”：

**2.1.1 类与实例的属性查找与可见性**

本节会推荐两个在工程化中“性价比最高”的特性：

- `@property` / `@<name>.setter`：

### 2.2 反射与自省

[返回目录](#课程目录)

本节对应 `we-know-python.md` 中的：

- [1.3.2 对象的属性 attribute](we-know-python.md#132-对象的属性-attribute)
- [2.2.4 面向对象的案例](we-know-python.md#224-面向对象的案例) 中的部分反射用法

结合大纲中“写更少 if-else 的命令分发表 / 插件式扩展点”，目标是：

- 让你掌握 `getattr` / `setattr` / `hasattr` 的**安全用法**；
- 用一两个实际例子展示“用反射代替长 if-elif 分支”的场景。

**2.2.1 常用自省 API**

- `dir(obj)` / `obj.__dict__`：查看有哪些属性；
- `getattr(obj, name, default)`：安全地获取属性；
- `setattr(obj, name, value)`：在运行时添加 / 修改属性。

**2.2.2 命令分发表：替代大 if-elif 的模式**

对比两种写法：

- 传统写法：`if cmd == "add": ... elif cmd == "remove": ...`；
- 基于映射表的写法：`handler_map[cmd](args)`；
- 再进一层，用反射：根据命令名称拼接方法名，如 `handle_<cmd>`，通过 `getattr` 动态分发。

这种模式会在：

- 日志分析工具的子命令；
- 智能助手的“子技能”路由；

中反复出现，有利于后续扩展新能力时只“加一个函数 / 类”，而不是改一大段 if-else。

### 3.1 Agent 是什么：概念、组成与实现

[返回目录](#课程目录)

在动手写代码之前，我们需要先搞清楚：**Agent 到底是什么**。这不仅关系到第三天的学习目标，也关系到你如何理解"AI 编程"这件事本身。

**3.1.1 两个容易混淆的概念**

首先要澄清一个常见误解：**“AI 大模型编程”，和“实现 Agent”**。

| 概念       | 含义                   | 例子                           |
| -------- | -------------------- | ---------------------------- |
| AI 大模型编程 | 用 LLM 帮你写代码（手工或自动生成） | Cursor、Copilot 帮你写 Python 函数 |
| 实现 Agent | 构建能自主感知、决策、行动的系统     | 智能日志分析助手、告警聚合助手              |

本课程第三天的目标是**融合两者**：

1. 手段：如何借助 AI 大模型完成**工程化的 Python 代码编写**？
2. 目的：如何用 Python 实现 Agent 系统？
3. 如何用 AI 大模型编程来帮你写 Agent 的代码。

**3.1.2 Agent 的起源与定义**

"Agent" 这个词在 AI 领域有悠久的历史：

- **词源**：来自拉丁语 _agere_，意为"行动、做"。
- **哲学传统**：亚里士多德的"代理因"（efficient cause），指能够自主行动、产生影响的实体。
- **AI 领域引入**：1980s-1990s，Russell & Norvig 在《Artificial Intelligence: A Modern
  Approach》（1995）中给出经典定义：

> **Agent 是一个能够感知环境并采取行动以实现目标的系统。**

这一定义沿用至今，包含四个核心要素：

```
Agent = 感知(Perception) + 决策(Decision) + 行动(Action) + 目标(Goal)
```

**强化学习视角（1990s-）**：Agent 是在环境中学习最优策略的学习器，通过试错最大化累积奖励。DeepMind 的 DQN、AlphaGo 都是这个传统。

**大模型时代（2020-）**：Agent = 预训练模型（决策核心）+ Harness（感知与行动环境）。Cursor、Claude Code 都是这种模式。

**3.1.3 现代 Agent 的组成模块**

一个完整的 Agent 系统通常包含以下模块：

```
┌─────────────────────────────────────────────────────┐
│                    Agent 系统                        │
│                                                     │
│  ┌─────────────┐         ┌─────────────┐          │
│  │   感知层    │────────▶│   决策层    │          │
│  │ Perception  │         │  Decision   │          │
│  │             │         │   (Model)   │          │
│  │ - 输入解析  │         │             │          │
│  │ - 状态理解  │         │ - 任务规划  │          │
│  │ - 知识检索  │         │ - 工具选择  │          │
│  └─────────────┘         │ - 异常处理  │          │
│                          └─────────────┘          │
│                                 │                  │
│                                 ▼                  │
│                          ┌─────────────┐          │
│                          │   行动层    │          │
│                          │   Action    │          │
│                          │             │          │
│                          │ - 工具执行  │          │
│                          │ - 结果整合  │          │
│                          │ - 输出生成  │          │
│                          └─────────────┘          │
│                                                     │
│  辅助模块：                                         │
│  - 记忆（短期/长期/RAG）                            │
│  - 工具注册表（Tools Registry）                     │
│  - 权限与安全（Permissions）                        │
│  - 日志与可观测性（Logging & Observability）        │
└─────────────────────────────────────────────────────┘
```

**核心模块说明**：

| 模块    | 作用             | 例子                               |
| ----- | -------------- | -------------------------------- |
| 感知层   | 理解用户意图、环境状态    | 解析用户问题、读取日志内容                    |
| 决策层   | 规划任务、选择工具、处理异常 | LLM 决定调用 `get_basic_stats`       |
| 行动层   | 执行工具、整合结果、生成输出 | 执行统计函数、生成分析报告                    |
| 记忆    | 保存对话历史、知识库检索   | 最近 N 轮对话、运维手册 RAG                |
| 工具注册表 | 管理可用工具的定义和路由   | `{"get_stats": get_basic_stats}` |
| 权限控制  | 限制危险操作、沙箱隔离    | 禁止删除文件、需要用户确认                    |

**3.1.4 Agent 的输入、输出与实现范式**

**输入（Input）**：

- 用户请求（自然语言问题或指令）
- 环境状态（日志文件、系统配置、数据库内容等）
- 可选：上下文信息（对话历史、知识库片段）

**输出（Output）**：

- 结构化结果（结论、数据、建议）
- 或行动执行结果（修改了文件、发送了通知等）

**典型实现范式（伪代码）**：

```python
def agent_loop(messages, tools, max_iterations=10):
    """
    Agent 主循环：感知 → 决策 → 行动
    
    messages: 对话历史（包含用户问题、系统提示、历史回答）
    tools: 可用工具列表（函数定义）
    max_iterations: 最大迭代次数（防止死循环）
    """
    for _ in range(max_iterations):
        # 1. 调用 LLM 进行决策
        response = llm.chat(messages, tools=tools)
        
        # 2. 判断是否需要工具调用
        if response.stop_reason != "tool_use":
            # 无需工具，直接返回结果
            return response.content
        
        # 3. 执行工具调用
        tool_results = []
        for tool_call in response.tool_calls:
            tool_name = tool_call.function.name
            tool_args = parse_json(tool_call.function.arguments)
            
            # 执行工具并收集结果
            result = execute_tool(tool_name, tool_args)
            tool_results.append({
                "tool_use_id": tool_call.id,
                "content": json.dumps(result)
            })
        
        # 4. 将工具结果加入对话，继续下一轮
        messages.append({"role": "assistant", "content": response.content})
        messages.append({"role": "user", "content": tool_results})
    
    return "达到最大迭代次数，任务未完成"

# 使用示例
messages = [
    {"role": "system", "content": "你是日志分析助手..."},
    {"role": "user", "content": "最近 1 小时的错误主要是什么？"}
]
tools = [get_basic_stats_def, search_error_patterns_def]

answer = agent_loop(messages, tools)
```

这个循环是所有 Agent 系统的核心，后续各节会在此基础上叠加更多能力。

**3.1.5 为什么需要 Agent**

| 需求   | 传统方案       | Agent 方案       |
| ---- | ---------- | -------------- |
| 自动化  | 脚本、规则系统    | 自主决策，适应变化      |
| 复杂任务 | 多个子系统拼接    | 单一 Agent 端到端完成 |
| 交互式  | 命令行、Web 表单 | 自然语言对话         |
| 可扩展  | 修改代码       | 添加新工具即可        |

Agent 的本质是：**让模型做决策，让代码做执行，让系统自主完成任务**。

**两种 AI 编程范式**

随着 AI 编程工具的成熟，出现了两种主流的编程方式：

**1. Vibe Coding（氛围编码）**

"Vibe Coding" 一词由 Andrej Karpathy（前 Tesla AI 总监、OpenAI 联合创始人）在 2025 年提出，指一种**以对话为主的编程方式**：

- **特点**：用自然语言描述需求，让 AI 生成代码，再通过对话迭代优化
- **流程**：描述想法 → AI 生成 → 看效果 → 继续对话调整 → 再生成
- **适合**：快速原型、探索性项目、个人项目、学习新技术
- **工具**：Cursor、Claude Code、Windsurf 等
- **缺点**：代码质量依赖 AI、可能产生冗余代码、不适合大型项目

```
用户："帮我写一个日志分析工具，能统计错误分布"
AI：生成代码...
用户："加上按时间过滤的功能"
AI：修改代码...
用户："再加个 Web 界面"
AI：继续添加...
```

**2. Spec Coding（规格编码）**

Spec Coding 是一种**以文档和规范驱动的编程方式**，强调先设计再实现：

- **特点**：先写详细的技术规格文档（Spec），再让 AI 按规格实现
- **流程**：编写规格 → AI 实现 → 代码审查 → 测试验证 → 迭代优化
- **适合**：企业项目、团队协作、需要高质量代码的场景
- **工具**：Cursor（Spec 模式）、Claude Code、专业 Agent 工具
- **优点**：代码质量可控、可追溯、便于团队协作和代码审查

```
用户编写 Spec：
"""
## 日志分析工具 Spec

### 功能需求
1. 解析 Nginx/Apache 访问日志
2. 统计状态码分布、Top IP、错误趋势
3. 支持按时间范围过滤

### 技术要求
- 使用 Python 3.10+
- 使用 dataclass 定义配置
- 单元测试覆盖核心逻辑
- CLI 入口：python -m logai --log <path>
"""

AI 按规格实现 → 用户审查 → 测试验证
```

| 维度   | Vibe Coding       | Spec Coding |

| 适用场景 | 快速原型、个人项目         | 企业项目、团队协作   |
| 代码质量 | 依赖 AI、可能冗余        | 可控、可追溯      |

| 审查难度 | 较高（需要理解 AI 生成的代码） | 较低（按规格验收）   |

> **建议**：探索初期适合用 Vibe Coding，团队合作工程实现适合用 Spec Coding。

**3.2.1 LLM API 的通用调用模式**

无论是在线商用模型，还是公司内部网关，对话接口通常包含：

- 基础元素：
  - 模型名称；
  - `messages`（角色 + 内容）或 `prompt`；
  - 可选参数：`temperature`、`top_p`、`max_tokens` 等。
- 基本调用流程：
  1. 组装请求体；
  2. 携带鉴权信息（API Key、JWT、公司 SSO 等）发起 HTTP / SDK 调用；
  3. 解析响应体，提取文本内容与可能的工具调用信息；
  4. 捕获异常、打日志、必要时重试。

课程会给出一个与大纲中 `llm_client.py` 类似的示例，强调：

- 在客户端内部处理重试和错误码，不把"网络细节"泄露给业务层；
- 日志中至少包含：请求 ID / 截断后的 prompt 预览 / 耗时 / 状态码。

**3.2.2 编写统一的 `LLMClient` 封装**

结合第二天的配置与异常处理，本节会引导你：

- 把 LLM 网关地址、API Key、超时时间等放入配置类；
- 使用一个轻量的 `LLMClient` 类对外暴露少量方法，例如：
  - `chat(messages: list[dict], **kwargs) -> str`
  - 后续可扩展 `chat_with_tools` / `embed` 等。

你会为这个类：

- 写至少 1–2 个基础单元测试（可使用 mock 替代真实网络调用）；
- 用统一的日志格式记录每一次调用，以便之后在"智能日志助手"中自食其果。

### 3.3 工具型 Prompt 设计

[返回目录](#课程目录)

本节对标大纲中的"角色设定 / 指令结构化 / 输出格式约束"，核心是让模型**像一个稳定的"函数"而不是聊天机器人**。

**3.3.1 角色设定与边界说明**

- 在 `system` 或开头 prompt 中，明确告诉模型：
  - 你是"日志分析助手 / 智能运维助手"；
  - 只能基于给定数据和工具结果回答，不要编造不存在的日志；
  - 回答时要区分"事实（从日志中得到）"和"推测（可能性结论）"。

**3.3.2 结构化指令：先做什么，再做什么**

课程会推荐一种"三段式"结构：

1. 明确输入：用户问题 + 日志统计结果 + 相关配置；
2. 工作步骤：如"先总结关键指标 → 再找异常模式 → 再给出建议"；
3. 输出要求：包含哪些字段、以什么形式呈现。

这种结构有助于：

- 和后续的工具调用参数解析对齐；
- 在必要时"用程序解析模型输出"，而不是只能人工阅读。

**3.3.3 输出格式约束**

根据你的应用需求，选择：

- Markdown 报告（适合人读）：分标题、小结、列表；
- JSON 结构（适合程序再处理）：例如 `{ "summary": "...", "top_issues": [...], "suggestions": [...] }`；

并在 Prompt 中明确：

- 只输出一种格式；
- 不要额外解释；
- 对无法回答的情况，用特定字段标记（例如 `"confidence": "low"`）。

### 3.4 工具定义与调用闭环

[返回目录](#课程目录)

这一节会把"Python 工具函数"与"LLM 工具调用能力"真正连接起来，对标大纲中的：

- 用 JSON Schema 描述工具；
- 解析模型返回的工具调用请求；
- 跑通"LLM 选择工具 → 执行 → 解释结果"的闭环。

**3.4.1 把 Python 函数描述成"工具"**

以日志分析场景为例，你会选择若干函数：

- `get_basic_stats(log_path, top_n)`
- `search_error_patterns(log_path, keyword)`

并为它们写出清晰的：

- `name`：例如 `"get_basic_stats"`；
- `description`：面向模型的自然语言说明；
- `parameters`：参数名 / 类型 / 必填与否 / 取值范围；

可以用 JSON Schema 或一个约定好的 Python 结构表示。

**3.4.2 解析与执行工具调用**

当模型输出一个"工具调用请求"时，你需要：

- 安全地解析出 `tool_name` 与 `arguments`（JSON 解析 + 基本校验）；
- 在一个"工具注册表"（字典）里找到对应的 Python 函数；
- 执行并捕获异常，把结果打回给模型（或直接整合到最终回答）。

本节会给出一个简单的"工具路由器"示例，展示如何：

- 记录每一次工具调用的参数与结果（方便事后分析和调试 Agent 行为）；
- 为后续的多轮工具调用做好接口设计。

### 3.5 记忆与 RAG 集成

[返回目录](#课程目录)

本节对标大纲中的"短期记忆 / 长期记忆 / 检索增强生成"，目标是：

- 让你的 Agent 能够"记住最近几轮对话"；
- 能在回答时参考"运维手册 / 历史故障案例"等外部知识库。

**3.5.1 短期记忆：最近 N 轮对话**

- 实现一个简单的对话历史结构（列表或队列），只保留最近 N 轮；
- 在发送给 LLM 的 `messages` 中，合理截断历史，避免上下文过长；
- 对于敏感信息，考虑在入库前做脱敏或过滤。

**3.5.2 长期记忆：RAG 管线**

不会深入讲全文检索理论，而是给出一个最小 RAG 流程：

1. 预处理文档（运维手册 / FAQ / 典型故障分析）并拆分成片段；
2. 使用 Embedding 模型将片段向量化，存入简单向量库（可选：Chroma、FAISS、Postgres 向量扩展等）；
3. 在每次问答前，根据用户问题检索若干最相似片段，拼接进 Prompt。

你会看到一个与课程大纲类似的例子，将：

- "运维手册 / FAQ" 作为知识库；
- 将检索到的片段附在用户问题后，让模型基于真实数据回答。

### 3.6 轻量 Agent 框架实践

[返回目录](#课程目录)

本节对标大纲中的"用 LangChain 或原生 Function Calling 构建单体 Agent"，在课程中会选用一条主线（如"原生 Function Calling + 轻量工具路由"或
LangChain 的简化用法），目标是：

- 把前面所有能力（LLM 客户端 / 工具描述 / 记忆 / RAG）组装成一个**可调用的 Agent 函数**；
- 支持多轮对话、自动选择工具，并产出结构良好的答案。

**3.6.1 单 Agent 主循环（简化版）**

我们不会在本课重写完整 Agent 主循环，而是用一个简化版本：

1. 组装当前对话上下文（用户问题 + 短期记忆 + RAG 结果）；
2. 调用 LLM 获取回复（可能包含工具调用意图）；
3. 如果需要工具调用：
   - 解析工具请求；
   - 调用 Python 工具；
   - 把工具结果填回对话，再调用一次 LLM 生成最终回答；
4. 更新对话历史与日志。

通过这个流程，你可以清楚看到：

- 哪些环节靠 Python 控制（确定性逻辑）；
- 哪些环节交给 LLM 决策（选择工具 / 生成自然语言报告）。

## 4. 实战具体方案

[返回目录](#课程目录)

### 4.1 第三天作业定义（推荐：告警聚合助手）

#### 4.1.1 输入（Input）

- 用户自然语言问题（例如：“最近 1 小时 5xx 异常升高的主要原因是什么？”）；
- 日志文件路径（可使用课程样例日志或学员自带脱敏日志）；
- 可选参数（时间范围、关键词、`top_n`）；
- 可选知识文本（运维 FAQ / 历史故障摘要，作为轻量检索材料）。

#### 4.1.2 功能（Function）

助手至少需要具备以下能力：

- 能根据用户问题选择合适的工具函数；
- 能执行日志统计与错误模式检索等基础工具；
- 能将工具结果回填给 LLM，生成结构化分析报告；
- 能在信息不足时明确说明缺失数据，不编造事实；
- 能记录关键调用日志（调用了哪个工具、参数是什么、是否成功）。

#### 4.1.3 输出（Output）

输出采用固定结构，建议至少包含：

1. 结论（发生了什么）；
2. 关键数据（状态码分布、Top IP、关键时间窗口等）；
3. 建议（优先排查步骤）。

当无法得出结论时，输出“信息不足”，并给出下一步需要补充的数据项。

#### 4.1.4 一天可交付范围（MVP）

为保证初学者可在 1 天内完成，建议将范围收敛为：

- 一个可用的 `LLMClient`（含超时与异常处理）；
- 2 个工具函数（例如 `get_basic_stats`、`search_error_patterns`）；
- 一条工具调用闭环（模型选工具 -> Python 执行 -> 回填结果 -> 生成回答）；
- 一个 CLI 入口（不强制 Web）；
- 2–3 个基础测试

### 4.2 逐步实现（教学与作业执行顺序）

下面给出推荐的“从零到可演示”实施路径。每一步都尽量保证“先可运行，再优化”。

#### 第 1 步：准备工程骨架与配置

- 建立最小目录结构（`config.py`、`llm_client.py`、`tools.py`、`agent.py`、`cli.py`）；
- 将模型地址、API Key、超时参数放入配置；
- 本地先跑通一次 “Hello LLM” 调用。

**验收标准**：执行脚本能返回一条模型文本回复。

#### 第 2 步：实现工具函数（先确定性逻辑）

- 在 `tools.py` 中实现日志基础统计与错误模式检索；
- 明确函数参数、返回结构和异常处理；
- 尽量输出结构化结果（字典而不是纯文本）。

**验收标准**：不接 LLM 时，工具函数可独立处理样例日志并返回结果。

#### 第 3 步：实现工具注册与调用路由

- 为每个工具定义 `name`、`description`、`parameters`；
- 建立“工具名 -> Python 函数”的注册表；
- 解析模型工具调用参数并执行对应函数。

**验收标准**：给定模拟的工具调用请求，能够正确路由并拿到执行结果。

#### 第 4 步：组装 Agent 闭环

- 构造消息上下文（用户问题 + 必要背景）；
- 第一次调用 LLM，获取是否需要工具调用；
- 若需要工具调用，执行后将结果回填，再次调用 LLM 生成最终答复；
- 对关键步骤输出日志，便于调试。

**验收标准**：至少跑通一次“提问 -> 调工具 -> 输出结论”的端到端流程。

#### 第 5 步：提供 CLI 交互入口

- 支持传入日志路径与用户问题；
- 支持循环提问（可选）；
- 在终端按“结论 / 关键数据 / 建议”格式输出。

**验收标准**：命令行可以连续提问并得到结构化答复。

#### 第 6 步：补最小测试与一页说明

- 为 `tools.py` 和路由逻辑各补至少 1 个测试；
- 编写一页技术说明，覆盖：
  - 用到的 Python 进阶特性；
  - 用到的 AI 能力点；
  - 后续扩展方向（RAG、更多数据源、安全审计）。

**验收标准**：`pytest` 通过基础用例，说明文档可独立阅读并复现项目目标。

#### 4.2.1 建议时间分配（6 小时）

- 2.0h：配置与 `LLMClient`；
- 1.5h：工具函数与路由；
- 1.5h：Agent 闭环与 CLI；
- 1.0h：测试、文档与演示彩排。

#### 4.2.2 超出 1 天范围的扩展项（可选）

- 接入向量库实现标准 RAG；
- 增加长期记忆与多轮会话管理；
- 提供 Web 页面与更完整的可观测性面板；
- 增加安全策略（敏感信息脱敏、审计日志、权限分级）。
